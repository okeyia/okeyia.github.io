一 Git学习

可视化 git 学习网站 Learn Git Branching

1.1 git checkout

用来切换分支, 表明当前所在的分支

在 Git2.23 版本中，引入了一个名为 git switch 的新命令，最终会取代 git checkout，因为 checkout 作为单个命令有点超载（它承载了很多独立的功能）

1.2 git merge

将其他的分支合并到当前的分支, 比如将bugFix 合并到main

1.3 git rebase

第二种合并分支的方法是 git rebase。Rebase 实际上就是取出一系列的提交记录，“复制”它们，然后在另外一个地方逐个的放下去。

Rebase 的优势就是可以创造更线性的提交历史，这听上去有些难以理解。如果只允许使用 Rebase 的话，代码库的提交历史将会变得异常清晰。

此时main还未更新, 使用下列操作更新main。

由于 bugFix 继承自 main，所以 Git只是简单的把 main 分支的引用向前移动了一下而已。

1.4 HEAD

HEAD 是一个对当前检出记录的符号引用 —— 也就是指向你正在其基础上进行工作的提交记录。

HEAD 总是指向当前分支上最近一次提交记录。大多数修改提交树的 Git命令都是从改变 HEAD 的指向开始的。

HEAD 通常情况下是指向分支名的（如 bugFix）。在你提交时，改变了 bugFix 的状态，这一变化通过 HEAD 变得可见。

分离的HEAD: 分离的 HEAD 就是让其指向了某个具体的提交记录而不是分支名。

1.5 撤销变更

Git里撤销变更的方法很多。和提交一样，撤销变更由底层部分（暂存区的独立文件或者片段）和上层部分（变更到底是通过哪种方式被撤销的）组成。我们这个应用主要关注的是后者。

主要有两种方法用来撤销变更 —— 一是 git reset，还有就是 git revert。

1.5.1 git reset

git reset 通过把分支记录回退几个提交记录来实现撤销改动。你可以将这想象成“改写历史”。git reset 向上移动分支，原来指向的提交记录就跟从来没有提交过一样。

原本的main 是指向c2的, 使用git reset HEAD~1后，main分支有迁回了c1，本地代码库就不知道c2的存在了。

1.5.2 Git Revert

虽然在你的本地分支中使用 git reset 很方便，但是这种“改写历史”的方法对大家一起使用的远程分支是无效的哦！

为了撤销更改并分享给别人，我们需要使用 git revert。

使用命令： git revert HEAD 进行撤销。

在我们要撤销的提交记录后面居然多了一个新提交！这是因为新提交记录 C2' 引入了更改 —— 这些更改刚好是用来撤销 C2 这个提交的。也就是说 C2' 的状态与 C1是相同的。

revert 之后就可以把你的更改推送到远程仓库与别人分享啦。

1.6 远程分支

远程分支反映了远程仓库(在你上次和它通信时)的状态。这会有助于你理解本地的工作与公共工作的差别 —— 这是你与别人分享工作成果前至关重要的一步.

远程分支有一个特别的属性，在你检出时自动进入分离 HEAD 状态。Git这么做是出于不能直接在这些分支上进行操作的原因, 你必须在别的地方完成你的工作, （更新了远程分支之后）再用远程分享你的工作成果。

1.7 git fetch

Git远程仓库相当的操作实际可以归纳为两点：

向远程仓库传输数据以及从远程仓库获取数据。既然我们能与远程仓库同步，那么就可以分享任何能被 Git管理的更新（因此可以分享代码、文件、想法、情书等等）。

执行 git fetch 的变化：

git fetch 并不会改变你本地仓库的状态。它不会更新你的 main 分支，也不会修改你磁盘上的文件。

1.8 git pull

然我们已经知道了如何用 git fetch 获取远程的数据, 现在我们学习如何将这些变化更新到我们的工作当中。

其实有很多方法的 —— 当远程分支中有新的提交时，你可以像合并本地分支那样来合并远程分支。也就是说就是你可以执行以下命令:

  • git cherry-pick o/main
  • git rebase o/main
  • git merge o/main
  • 等等

实际上，由于先抓取更新再合并到本地分支这个流程很常用，因此 Git提供了一个专门的命令来完成这两个操作。它就是我们要讲的 git pull。

  1. 使用命令 git fetch; git merge o/main

    我们用 fetch 下载了 C3, 然后通过 git merge o/main 合并了这一提交记录。现在我们的 main 分支包含了远程仓库中的更新（在本例中远程仓库名为 origin）

  2. 使用 git pull 可以达到同样的效果

1.9 git push

git push 负责将你的变更上传到指定的远程仓库，并在远程仓库上合并你的新提交记录。一旦 git push 完成, 你的朋友们就可以从这个远程仓库下载你分享的成果了！

*注意 —— git push 不带任何参数时的行为与 Git的一个名为 push.default 的配置有关。它的默认值取决于你正使用的 Git的版本，但是在教程中我们使用的是 upstream*。

使用 git push 后的变化：

远程仓库接收了 C2，远程仓库中的 main 分支也被更新到指向 C2 了，我们的远程分支 (o/main) 也同样被更新了。所有的分支都同步了！

二 ssh-key 配置

2.1 github 使用 token

为了安全起见，github不再支持密码登录，改为了token，token就相当于一个比较安全的密码，获取到的token是全局的，可以对所有仓库进行操作

  1. 获取token，在个人设置下，Developer settings中，新增token

  2. 拿到token后，有两种使用方式

    • 使用git的时候，每次输入账号密码，这里的密码就是token，但是每次使用都要输入，比较麻烦

    • 在git中的远程分支添加token

    • 然后就可以愉快的 git push origin master了

2.2 github 添加私钥

这里的话配置的是 某个仓库 的私钥，只允许对某个仓库进行操作。第 2.1 章节 配置的全局的私钥，可以对账号下面的所有仓库进行操作。

建议新建一个demo的仓库，在demo中进行配置

  1. 在终端中生成一个 github 的 ssh-key

  1. 将私钥对应的公钥,也就是 github_id_rsa.pub 的内容copy到github中某个仓库

  2. 按照第2.3章节，配置 config 文件

  1. 配置完成后, 测试一下

2.3 为主机指定私钥

主机上可能存在多个私钥, 但是连接的时候默认使用的是id_rsa, 如果不匹配的话, 则连接失败。有三种方法实现为主机指定密钥：

  • ssh -i xxxx 用户@ip地址，（这里xxx是指密钥文件），这样的话每次都要输入密钥文件，比较麻烦

  • 将密钥交给 ssh-agent 进行管理，但是ssh-agent在系统中并没有开启自启，每次都需要手动开启，不推荐

  • 配置 config 文件，一劳永逸，强烈推荐

SSH的配置文件有两个:

常用的配置参数:

  • Host

    用于我们执行 SSH 命令的时候如何匹配到该配置。

    • *，匹配所有主机名。
    • *.example.com，匹配以 .example.com 结尾。
    • !*.dialup.example.com,*.example.com，以 ! 开头是排除的意思。
    • 192.168.0.?，匹配 192.168.0.[0-9] 的 IP。

  • HostName

    真实的主机名，默认值为命令行输入的值（允许 IP）

  • Port

    默认端口为22, 使用其他端口的话加上 -p xxx

  • User

    登录的用户名

  • IdentityFile

    指定连接的私钥

3.4 通配符

常用的通配符有：

（1）星号“*” ：匹配多个字符；

（2）问号“?”：匹配除 ‘/’外的任意一个字符；

（3）方括号“[xxxx]”：匹配多个列表中的字符；

3.5 其他规则

  1. 空行不匹配任何文件；
  2. git 跟踪文件，而不是目录；
  3. 在 .gitignore 文件中，每行表示一种模式；
  4. 如果本地仓库文件已被跟踪，那么即使在 .gitignore 中设置了忽略，也不起作用。这个时候使用 git rm -rf --cached . 先将git cache中的内容全部删除, 然后再重新添加。
  5. .gitignore 文件也会被上传的到远程仓库，所以，同一个仓库的人可以使用同一个.gitignore 文件。
